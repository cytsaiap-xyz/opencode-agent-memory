# AI Agent Memory Systems — Landscape Report (July 2026)

**Purpose:** architecture selection for a self-hosted engineering-knowledge memory system.
**Deployment constraints:** fully on-prem (semiconductor IC-design; data cannot leave premises), self-hosted vLLM with OpenAI-compatible API, TypeScript+Bun / Python+uv stacks, must integrate with an existing markdown wiki, batch/scheduled consolidation acceptable.

---

## 0. Executive Summary

Three findings dominate the mid-2026 landscape:

1. **The field has converged toward file-based memory from two directions.** Claude Code (markdown MEMORY.md index + topic files, explicit no-RAG design), Anthropic's API memory tool (client-side file ops), Letta's 2026 pivot to "MemFS" (git-backed markdown memory with dream-agent consolidation), and MemU (compiles memory to INDEX.md/MEMORY.md/SKILL.md) all landed on *plain files + index + agentic lexical search*. Letta's own benchmark showed a grep-over-files agent (74.0% LoCoMo) beating Mem0's graph variant (68.5%). Engineering content — terminology-dense, exact-match-friendly — is precisely the regime where lexical search wins.
2. **The OSS/commercial split widened in 2025-2026.** Zep Community Edition was frozen (Apr 2025, all OSS effort moved to Graphiti); Mem0's OSS v3 (Apr 2026) *removed* graph memory and UPDATE/DELETE consolidation, keeping them Platform-only; Supermemory's real on-prem story is enterprise-sales-gated; Letta deprecated its self-hosted server surface mid-flight. Vet governance trajectory, not just current license.
3. **The universal on-prem risk is structured-output fidelity of local models.** Every extraction-pipeline system (Graphiti, Cognee, Mem0, Hindsight) is gated on schema-compliant JSON from your LLM. vLLM's guided decoding (`response_format: json_schema`) with a ≥14-32B instruct model mitigates this; spike-test extraction quality before committing to any pipeline system.

**Verdict preview:** for this deployment, the strongest architecture is a **markdown-wiki-as-source-of-truth core** (the pattern already validated by Claude Code / Letta MemFS), optionally augmented by **Graphiti** (temporal relations/supersession queries) or **Cognee** (graph+vector with ontology) as a *rebuildable derived index* — never as the system of record. Details in §4.

## 2. Per-System Detail

### 2.1 Mem0 (mem0ai)

- **Status:** ~60.5k★, Apache-2.0, $24M Seed+Series A (Oct 2025, YC/Kindred/Basis Set). Python SDK 2.0.11 / TS SDK 3.0.13 (Jul 2026).
- **Classic architecture (the paper, arXiv:2504.19413):** two-phase pipeline — (1) LLM extracts atomic facts from rolling summary + recent messages; (2) each fact embedded, compared to existing memories, second LLM call routes to **ADD / UPDATE / DELETE / NOOP**. Vector store default Qdrant (20+ backends). **Mem0ᵍ** graph variant added entity/relation triples — only +1.6pt on LoCoMo (68.4 vs 66.9), mainly helping temporal/multi-hop.
- **Critical 2026 change — OSS v3 algorithm (Apr 2026):** single-pass **ADD-only** extraction (consolidation call removed from OSS); hybrid retrieval (semantic + BM25 + entity-linking boost); **external graph store support deleted from OSS** (Neo4j/Memgraph/Kuzu integrations removed, PR #4805) — replaced by entity-collections used only for ranking boosts, not traversable relations. Real graph memory and full consolidation are now **Platform-only**; the README admits published benchmarks reflect the managed platform's "proprietary optimizations."
- **Self-hosting:** genuinely first-class — `openai_base_url` on both LLM and embedder, dedicated vLLM provider, embedded Qdrant/SQLite library mode or Docker server mode with pgvector. **Disable default-on telemetry** (`MEM0_TELEMETRY=false`) for air-gapped use.
- **MCP:** OpenMemory MCP (May 2025) — Dockerized FastAPI + Qdrant + dashboard; tools: `add_memories`, `search_memory`, `list_memories`, `delete_all_memories`. Default config expects OpenAI keys; local-endpoint wiring needs overrides (community forks exist).
- **For engineering know-how:** the fundamental critique is **lossy extraction** — write-time atomization discards the *why* and reasoning chain ("prefers Qdrant" without context). Default prompts target personal preferences; overriding requires `custom_instructions`. With OSS v3, stale engineering facts (superseded workarounds) **accumulate with no UPDATE/DELETE path** — you'd write your own batch consolidation job. Also note the Mem0-vs-Zep LoCoMo benchmark war (each side reproducing the other 10-25pts lower); treat all vendor LoCoMo numbers skeptically — the benchmark itself is flawed.
- **Fit:** best-in-class TS+Python parity and community scale, but the OSS trajectory (features removed, best pipeline paywalled) is the wrong direction for a long-lived on-prem bet, and the fact shape is wrong for engineering lessons.

### 2.2 Letta (formerly MemGPT)

- **Status:** in the middle of a **major March 2026 pivot** ("Letta's Next Phase"). Two Lettas now exist:
- **Classic server (legacy-ish):** self-editing in-context **memory blocks** (persona/human/custom, sharable across agents), **recall memory** (full history in Postgres), **archival memory** (pgvector passages, agent-driven insert/search), **sleep-time compute** (paired background agent consolidates memory during idle). Docker + Postgres+pgvector; vLLM supported via `OPENAI_API_BASE` (requires tool-calling support). **But the docs now state the self-hosted server "is no longer an actively maintained or supported product surface."** v0.16.8, 23.7k★, Apache-2.0. Legacy memory tools and server-side sleep-time agents deprecated on a weeks-long timeline in 2026.
- **Letta Code / MemFS (2026 flagship):** TypeScript CLI/harness (2.8k★, v0.28.0). Memory is a **git repo of markdown files** — `system/` files always in prompt, rest indexed in a memory tree loaded on demand; every memory change is a git commit; subagents write via worktrees; **"dream" subagents batch-review recent sessions and write lessons into memory files**. Model-agnostic, but Letta's own docs warn the harness "is very demanding" and underperforms with most open-weight models. Cloud sync ("Constellation") must be avoided for on-prem.
- **MCP:** Letta is primarily an MCP *client*; exposing Letta memory as MCP is community-only (oculairmedia/Letta-MCP-server).
- **For engineering know-how:** memory blocks = self-editing CLAUDE.md (good for doctrine); archival = flat vector RAG (no relations/temporal validity). **MemFS is the single best architectural idea in the survey for this context** — git-backed markdown memory with scheduled consolidation agents maps 1:1 onto a markdown wiki. But **platform churn is the top risk**: the API surface you'd have built on in 2025 was deprecated in 2026.
- **Fit:** borrow the MemFS pattern; don't adopt Letta as system-of-record.

### 2.3 Zep / Graphiti

- **Zep CE is dead:** frozen April 2025 (code moved to `legacy/`, no updates; not relicensed). Zep platform is cloud-first commercial (VPC/BYOC via enterprise sales only) — **ruled out by the on-prem constraint**. All OSS effort went to **Graphiti** (28.6k★, Apache-2.0, v0.29.2 Jun 2026, very active; promoted by Neo4j itself).
- **Architecture:** you ingest **episodes** (message/text/JSON); LLM pipeline extracts entities + relation facts, dedupes against the graph, resolves contradictions; every fact retains **provenance to its source episode**. **Bi-temporal model:** each fact edge carries a validity window plus ingestion time; contradicted facts are **invalidated (dated), never deleted** — point-in-time queries work ("what did we believe at tape-out X"). Three subgraph tiers (episode → entity → community). **Retrieval is hybrid semantic + BM25 + graph traversal with rerankers, no LLM call at query time** — sub-second and cheap on GPU budget. **Custom ontology via Pydantic models**: define `Bug`, `Decision`, `Tool`, `Process` entities and `SUPERSEDES`, `CAUSED_BY` edges.
- **Self-hosting:** first-class. Graphiti + Neo4j 5.26+ or FalkorDB (incl. embedded FalkorDB-Lite); bundled FastAPI REST server for non-Python clients; `OpenAIGenericClient` explicitly supports vLLM/Ollama/llama.cpp with `structured_output_mode: "json_schema"` (use vLLM guided decoding) or `"json_object"`. Telemetry opt-out. `SEMAPHORE_LIMIT` tunes concurrency against vLLM throughput.
- **Known risk:** extraction is entirely gated on structured-output fidelity — small models produce schema-invalid JSON (issues #868, #796). vLLM constrained decoding + a ≥14-32B model is the recommended configuration; **spike this first**.
- **Language/MCP:** Python-only core (uv-friendly); TS integrates via REST or the **first-party MCP server** (add_memory/episodes, `search_facts`, `search_nodes`, `get_episodes`, delete/clear; stdio + HTTP; Docker Compose with FalkorDB default).
- **For engineering know-how:** the bi-temporal invalidate-don't-delete model is the best match in the survey for decision supersession and audit-minded environments. Weakness: multi-paragraph debugging narratives get shredded into atomic facts — keep the wiki as the narrative layer, Graphiti as the relations/index layer.

### 2.4 LangMem (LangChain)

- **Status:** alive but semi-dormant — repo active into June 2026 but latest release **0.0.30 (Oct 2025)**, pre-1.0, MIT, 1.5k★, ~746K monthly downloads. Not deprecated, not absorbed; LangChain's energy shifted to LangGraph-native memory.
- **Architecture:** an SDK of primitives, not a server. **Hot path**: `create_manage_memory_tool` / `create_search_memory_tool` (agent decides). **Background**: `create_memory_manager` / `create_memory_store_manager` driven by **`ReflectionExecutor`** (debounced, processes after idle) — the manager LLM inserts/updates/deletes memories **against typed Pydantic schemas**. Memory types: semantic, episodic, procedural (prompt optimizer). Storage: LangGraph BaseStore → **Postgres+pgvector** with namespaced tuples (natural per-project scoping); `embed` accepts an arbitrary function.
- **Self-hosting:** trivial — pip library, no SaaS, no telemetry; `ChatOpenAI(base_url="http://vllm:8000/v1")` works directly.
- **Weaknesses:** **Python-only** (no TS SDK); **no MCP server** (DIY wrap); maintenance-risk; no graph.
- **For engineering know-how:** **the typed-schema idea is the killer feature** — define `DebuggingLesson{symptom, root_cause, fix, project}`, `ArchDecision{context, decision, alternatives}` and get schema-controlled extraction and consolidation, with clean markdown serialization. Best consumed as an architectural pattern (or a small dependency) rather than a platform.

### 2.5 Cognee

- **Status:** 27.4k★, **Apache-2.0**, $7.5M seed (Feb 2026, Pebblebed), **Cognee 1.0 released June 26, 2026** — API of `remember / recall / improve / forget`, single-Postgres deployment mode, official **TypeScript and Rust SDKs**. Very active (releases into July 2026).
- **Architecture (ECL pipeline):** Add → **Cognify** (classify → chunk → **LLM triplet extraction**, schema emergent unless constrained → summarize → embed DataPoints) → hybrid **graph + vector + relational metastore**. **Incremental** — already-processed items skipped (good for nightly wiki-delta syncs). **Memify** (`improve`) is the batch consolidation pass: prune stale nodes, reweight edges by usage, derive new facts. **15 search modes**: GRAPH_COMPLETION (default), multi-hop COT, HYBRID (BM25+vector+graph), TEMPORAL, raw CYPHER, NL→Cypher, CODE, plus retrieval-only modes. **OWL/RDF ontology support** validates extracted entities against a curated schema — directly useful for a `Component/Bug/RootCause/Fix/Decision` vocabulary. Custom typed DataPoints via Pydantic.
- **Backends:** graph Kuzu (embedded) / Neo4j / Postgres-native; vector LanceDB (embedded) / pgvector / Qdrant / Chroma / Milvus / Weaviate; metastore SQLite/Postgres. Zero-extra-infra local mode or **everything-in-one-Postgres** prod mode.
- **Self-hosting:** `LLM_PROVIDER=custom` + `LLM_ENDPOINT` — **vLLM explicitly supported**; embeddings configured separately. **Gotcha: configure both `LLM_*` and `EMBEDDING_*` or the unset one silently defaults to api.openai.com** — a hard failure mode for air-gapped deployment. Cognify is LLM-call-heavy; community testing shows small local models fail on *schema compliance*, not comprehension — use guided decoding + the strongest available model.
- **MCP:** first-party `cognee-mcp` (~11 tools): `cognify`, `codify` (repo → code graph), `search`, `save_interaction`, `cognee_add_developer_rules`, `list_data`, `delete`, `prune`, background-job status. stdio/SSE/HTTP.
- **For engineering know-how:** graph relations are the natural shape of engineering knowledge (`Bug —caused_by→ RaceCondition —fixed_by→ CommitX`); multi-hop COT answers "why did we choose X over Y." Weaknesses: emergent-schema noise (use strict prompt + ontology from day one); memify's derived facts are LLM inferences — treat as reviewable, not authoritative; API churn risk (1.0 is weeks old — pin versions); no usage-based salience decay.

### 2.6 Academic Approaches (policy donors, not deployables)

- **MemoryBank** (AAAI 2024) — hierarchical storage (raw log → daily summary → global profile) + **Ebbinghaus forgetting curve** (recall reinforces, disuse decays `e^(−t/S)`). Code frozen since 2023 (438★). **Lesson:** hierarchical summarization maps to "session transcript → daily dev journal → project profile"; recurrence-based *reinforcement* is right for pitfalls, but **pure time-decay is wrong for engineering facts** — a rarely-accessed root-cause analysis remains true. Decay salience/ranking, never delete.
- **A-MEM** (NeurIPS 2025, arXiv:2502.12110) — **Zettelkasten agentic memory**: atomic notes with LLM-generated keywords/tags/context; LLM-decided link generation; **memory evolution** — new notes trigger rewrites of linked older notes. ~915★ research code (ChromaDB). **Lesson:** validates the cross-referenced-pitfalls wiki pattern; retroactive evolution is exactly what "new fix supersedes old workaround" needs — but for engineering truth, **append revisions with an audit trail, never overwrite** (uncontrolled LLM rewriting corrupts ground truth).
- **Generative Agents** (Park et al., 2023; 21.7k★) — memory stream + **three-factor retrieval (recency × importance × relevance)** + **reflection trees** (periodic batch pass derives higher-level insights with citations to evidence memories). The most influential design; borrowed by nearly everything above. **Lesson:** hand-written pitfall lists are literally manual reflections — an automated nightly reflection pass over session logs/commits can propose new pitfall classes with evidence citations. Anchor "importance" to objective signals (severity, time lost, recurrence count), not LLM vibes.
- **Also shaping the field:** **MemOS** (MemCube: plaintext/KV-cache/LoRA memory — uniquely relevant when you control the vLLM serving stack, but operationally heavy and research-grade); **MIRIX** (six-type taxonomy — Core/Episodic/Semantic/**Procedural**/Resource/**Knowledge Vault** — the right *categories* for engineering memory, thin adoption); the Dec 2025 **47-author survey** (arXiv:2512.13564) whose "experiential memory" category is exactly this project; benchmarks **LoCoMo** (flawed, gamed by vendors) and **LongMemEval/-V2** (better — its *knowledge-update* and *abstention* categories are the failure modes that matter for engineering correctness).

### 2.7 Simple File-Based Approaches

- **Claude Code auto-memory** — per-repo `~/.claude/projects/<project>/memory/`: **MEMORY.md index (only first 200 lines / 25KB auto-loaded)** + topic files read on demand with ordinary file tools; Claude writes selectively mid-session; fully auditable/git-movable. Plus the CLAUDE.md hierarchy (managed/user/project/local, `@imports`, path-scoped `.claude/rules/*.md`). Anthropic's API-level **memory tool** (`memory_20250818`) is the same idea as a client-side primitive — Anthropic reports +39% on agentic-search evals and 84% token savings combined with context editing.
- **Markdown + grep/BM25** — Boris Cherny confirmed Claude Code dropped RAG + vector DB because "agentic search generally works better… simpler, without the security, privacy, staleness, reliability issues." **Letta's "Is a Filesystem All You Need?" benchmark: grep+file tools scored 74.0% LoCoMo (GPT-4o-mini), beating Mem0's graph variant (68.5%)** — simpler tools are better-represented in training data. Caveat: vector RAG regains ground at large corpus scale; a team wiki (thousands of pages) is comfortably below the crossover, especially for terminology-dense engineering text (signal names, error strings, tool flags) where lexical match dominates.
- **basic-memory** (Basic Machines) — v0.22.1, 3.4k★, **AGPL-3.0**, Python/uv. Markdown files are source of truth (Obsidian/git compatible) with a light semantic layer (`- [category] observation #tag`, `[[wikilinks]]`) parsed into a local SQLite FTS + local-embedding (FastEmbed ONNX, no external calls) hybrid index; MCP tools include graph-traversal `build_context`. No server-side LLM at all. Closest off-the-shelf match to "markdown wiki + knowledge graph + MCP." Caveats: AGPL (fine for internal use), single-user orientation (team concurrency = git DIY).
- **Git-vault reference implementations** (obsidian-mind, EchoVault, obsidian-agent-memory-skills) — young repos; the *pattern* (git for versioning/merge/audit, wikilinks for graph, FTS5/ripgrep for retrieval, session hooks for load/write-back) is what matters.

### 2.8 Notable New Entrants (2025-2026)

- **Hindsight (Vectorize)** — 18.2k★, **MIT**, v0.8.4. The strongest engineered pipeline among new entrants: **Retain** (extraction → entities/relations/time-series, sparse+dense vectors) / **Recall** (4 parallel strategies — semantic, BM25, graph, temporal — RRF-fused + cross-encoder rerank) / **Reflect** (offline batch pass forming "mental models" — productized batch consolidation). All in **PostgreSQL**; one-command Docker; `HINDSIGHT_API_LLM_BASE_URL` for OpenAI-compatible endpoints (known quirk: vLLM with `--enable-auto-tool-choice` silently downgrades `tool_choice="required"`, issue #1877). LongMemEval SOTA claim **independently reproduced** (Virginia Tech, Washington Post) — rare in this space. Weakness: memory lives in Postgres, not markdown — wiki sync is your work.
- **Memori (GibsonAI)** — 15.6k★, Apache-2.0. "Memory should be a SQL database you can inspect": BYODB (SQLite/Postgres/MySQL), **full-text retrieval, no vector DB**, ~1.3k tokens/query, MCP for Claude Code. Maximally auditable/greppable; but flat facts, no reflection layer.
- **Supermemory** — 28.3k★, MIT core, big benchmark claims, TS stack; but genuine air-gapped self-hosting is **enterprise-agreement-gated**; local single binary is v0.0.3. Friction for a data-cannot-leave-premises house.
- **MemOS (MemTensor)** — 10.2k★, Apache-2.0, v2.0.23. Memory-as-OS with **MemCube** (plaintext / KV-cache / LoRA memory with lifecycle, permissions, provenance); explicit `vllm` provider. The only project whose research direction (activation memory) directly exploits control of the serving stack — but heavy (Neo4j+Qdrant), DB-resident content, companion-ecosystem gravity. Watch-list.
- **Memobase** — profile-slot memory for companion apps; **wrong domain shape** for engineering knowledge. Skip.
- **Beads (Yegge)** — 25.2k★ in ~9 months, MIT, Go. Not a knowledge store: a **dependency-aware git/Dolt-backed issue graph as agent working memory**, built explicitly to cure "markdown dementia" (piles of conflicting plan files). `bd prime` injects context at session start; `bd remember` stores insights; compaction summarizes old closed work. **Complementary** — covers WIP/decision-trail memory, not distilled know-how.
- **mem-agent (Dria)** — a 4B Qwen3 model RL-trained to *operate* an Obsidian-like markdown memory via tool calls, rivaling 235B models on their memory bench. Research artifact, but a proof that **a small model on your own vLLM can be the memory gardener**, reserving the big model for engineering work.
- **MIRIX / MemU** — taxonomy and pattern donors respectively (six memory types; compile-everything-to-markdown). Monitor, don't adopt.

## 3. Cross-Cutting Findings

1. **Extraction-pipeline memory is lossy by construction.** The LLM decides at write time what will matter at read time; the *why* and reasoning chains — the core of engineering know-how — are exactly what atomization discards. Every pipeline system needs custom prompts/schemas to escape personal-assistant fact shapes; only LangMem (Pydantic schemas), Graphiti (Pydantic ontology), and Cognee (OWL ontology + DataPoints) make this first-class.
2. **Wiki-as-source-of-truth, engine-as-derived-index is the safe topology.** Both Cognee and Graphiti graphs are rebuildable projections of ingested text. Keeping canonical knowledge in git-backed markdown (a) de-risks vendor/API churn (Letta, Mem0 both broke users in 2026), (b) keeps human curation and audit trails, (c) preserves narrative context the graphs shred.
3. **Batch consolidation is well-supported everywhere it matters.** Cognee memify, Hindsight Reflect, LangMem ReflectionExecutor, Letta dream agents, Generative-Agents-style reflection — the "batch OK, no realtime needed" constraint costs nothing and rules nothing in.
4. **Benchmarks are marketing.** The Mem0-vs-Zep LoCoMo war (each reproducing the other 10-25pts lower), LoCoMo's design flaws, and platform-only "proprietary optimizations" behind headline numbers mean: evaluate on **your own wiki** with LongMemEval-style categories — especially *knowledge updates* (new fix supersedes old workaround) and *abstention* (don't invent a lesson).
5. **The one hard spike to run before any commitment:** structured-output extraction quality of the best vLLM-served model (guided decoding, `json_schema`), field-by-field validated — consistent with the existing house rule on schema-validating LLM output.

## 4. Fit Assessment & Recommendation

For a fully on-prem, vLLM-backed, markdown-wiki-integrated engineering-knowledge system:

**Tier 1 — Foundation (adopt):** the **file-based pattern already in use**, hardened. Git-backed markdown wiki as system of record; MEMORY.md-style index (≤200 lines) + one-topic-per-file; ripgrep/BM25 agentic retrieval; a scheduled consolidation job (nightly reflection pass à la Generative Agents: dedupe, mark `SUPERSEDES`, propose new pitfall classes with evidence citations; A-MEM-style revisions appended with audit trail, never overwritten; recurrence-count reinforcement à la MemoryBank instead of time-decay deletion). This is validated by Claude Code's design, Letta's MemFS pivot, Anthropic's memory tool, and the Letta filesystem benchmark — and it is the *only* approach with zero wiki-integration cost. The consolidation agent can be a small model on vLLM (mem-agent proof-of-concept).

**Tier 2 — Structured index on top (evaluate one, as a rebuildable projection):**
- **Graphiti + FalkorDB** if the priority is *temporal supersession and provenance* ("what did we believe at tape-out X; what superseded this workaround") — Apache-2.0, active, first-party MCP, explicit vLLM support, custom Pydantic ontology, no query-time LLM. Python-only core (TS via REST/MCP).
- **Cognee** if the priority is *multi-hop relational recall with ontology and richer search modes* — Apache-2.0, funded, 1.0 with TS SDK, single-Postgres mode, incremental cognify suits nightly wiki-delta syncs. Younger API; watch the silent-OpenAI-fallback gotcha.
- **Hindsight** if you'd rather buy a complete engine (MIT, Postgres-only, independently reproduced benchmarks, batch Reflect) and accept DB-resident memory + a DIY wiki bridge.

**Borrow, don't adopt:** LangMem's typed-Pydantic-schema extraction (`DebuggingLesson`, `ArchDecision`) as the schema for the consolidation job; MIRIX's memory taxonomy (procedural vs experiential vs vault); Beads for the *separate* problem of agent task/decision-trail state.

**Avoid for this use case:** Mem0 OSS (v3 removed exactly the consolidation and graph features engineering memory needs; wrong fact granularity), Letta as platform (2026 churn; classic server officially unmaintained — but steal the MemFS pattern), Zep platform (no self-serve on-prem; CE frozen), Supermemory (on-prem enterprise-gated), Memobase (person-centric), MemOS/MIRIX/MemU (research-watch).

**Pre-commitment spike (one afternoon):** point Graphiti or Cognee at 20 representative wiki pages via vLLM guided decoding with the best instruct model; hand-score extraction against a golden set of 15 questions spanning multi-hop, temporal-supersession, and abstention. If extraction quality fails, Tier 1 alone — which the 2026 evidence says is already a competitive memory system — is the answer.
